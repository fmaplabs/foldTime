# Stage 4: `db.rs` — SQLite storage

First third-party crate with real state: `rusqlite`. Schema, pragmas, and the
three functions everything else in the app calls.

## Concepts

- Integrating a third-party crate and its own `Result`/error type
- Writing and executing SQL from Rust
- Prepared statements
- Mapping rows to structs

## Tasks

- [x] Write the `heartbeats` table DDL and both indexes exactly as specified
  in the plan's Data Model section (`idx_heartbeats_project_task_ts`,
  `idx_heartbeats_commit_hash`)
- [x] `open_db(path: &Path) -> rusqlite::Result<Connection>`: opens/creates
  the file, runs the DDL (`CREATE TABLE IF NOT EXISTS`), sets `PRAGMA
  journal_mode=WAL`, sets a `busy_timeout`
- [x] `insert_heartbeat(&Connection, ...)`
- [x] `query_heartbeats(&Connection, project, task, since, until) -> rusqlite::Result<Vec<Heartbeat>>`,
  ordered by `ts ASC` (cheap thanks to the composite index)
- [x] `tag_untagged_heartbeats(&Connection, project, task, commit_hash)` —
  `UPDATE ... WHERE commit_hash IS NULL`
- [x] Unit tests via `Connection::open_in_memory()`: insert-then-query
  round-trip; tagging only touches previously-`NULL` rows; result ordering is
  `ts ASC`
- [x] Note: `PRAGMA journal_mode=WAL` is meaningless for an in-memory
  connection — don't let that assertion silently pass without meaning
  anything; if you want to actually verify WAL mode, do it against a
  `tempfile`-backed file connection in a separate test
